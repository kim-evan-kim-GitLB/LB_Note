## 3. 구현한 보안 통제 (영역별 상세)

### 3.1 인증 / 인가

- **JWT 기반 로그인**: `POST /api/auth/login` -> `{token, user}`, `GET /api/auth/me`(Bearer).
  PyJWT HS256, 서명 키 `JWT_SECRET`(.env). `JWT_SECRET` 미설정 시 부팅 실패(안전 기본).
- **비밀번호 해시**: passlib `pbkdf2_sha256`(순수 파이썬, bcrypt 백엔드 의존 회피). 평문 저장 없음.
- **전 엔드포인트 보호**: `/api/ai/*`, `/api/meetings*`, `/api/settings/*` 전부 의존성으로 보호.
  토큰 없음/위조/만료는 401.
- **회의 소유자 격리**: 토큰의 사용자로 owner 강제 주입.
  - list = 내 회의만, create = 클라이언트가 보낸 ownerId 무시하고 강제 주입
  - get/patch/delete = 소유 확인 후만, 남의 것은 404로 은닉(존재 여부 노출 차단)
  - patch 시 ownerId 변경 차단
- **계정 정책**: 자가가입 엔드포인트 없음. env(`WEB_AUTH_USERS`)가 단일 진실원천,
  `WEB_AUTH_ADMINS` 만 role=admin, 그 외 developer. `WEB_AUTH_PRUNE=0`로 일괄등록 계정 보존.
- 관련 파일: `src/web/auth.py`, `src/web/app.py`, `src/web/store.py`, `scripts/seed_users.py`

### 3.2 초기 비밀번호 강제 변경 (`ab5d654` + 프론트 `a60ad20`)

공용 초기 비번(예: 참가자 `axlead1234`, admin/dev 시드 비번)을 바꾸지 않고 방치하는 위험을 차단.

- DB `must_change_password` 컬럼 추가(기존 행은 마이그레이션으로 DEFAULT 1 = 변경 필요).
- 관리자/스크립트가 비번을 (재)설정하는 경로는 `must_change_password=1` 강제.
- 사용자가 셀프로 비번 변경(`POST /api/auth/change-password`) 시 `=0` 해제 -> 게이트 통과.
- `require_user_active` 게이트: 데이터/AI/설정 엔드포인트는 강제변경 대상이면 403
  (`error_code=must_change_password`). 단 `me`/`change-password`/`login`/`health`는 비차단(잠김 방지).
- 프론트: `/api/auth/me`의 `mustChangePassword` 플래그로 `ForcePasswordChangeView` 게이트,
  변경 즉시 해제(재로그인 불필요), 강제변경 대상은 회의 목록 로드 스킵(403 회피).
- **운영 영향(의도)**: 배포 후 전원 차회 로그인에 비번 변경이 강제된다.

### 3.3 자격증명 at-rest 암호화 (`ab5d654`, `5d11a2b`)

사용자별 Claude API키/구독토큰을 DB에 평문 저장하던 것을 Fernet 대칭암호로 보호.

- `CRED_ENC_KEY`(Fernet 키, base64 32B) 설정 시 저장값 암호화, 접두사 `enc:fernet:` 로 태깅.
  - 태깅으로 평문/암호문/키불일치를 결정적으로 구분 -> 키 교체/불일치 시 이중암호화 비가역 손상 방지.
  - `CRED_ENC_KEY` 미설정 시 평문 저장(무중단 하위호환), 설정하면 부팅 시 평문분 자동 재암호화 마이그레이션.
- 시크릿은 API 응답/로그에 노출하지 않음(상태만 노출). PUT 시 검증 ping으로 유효성 확인.
- 사용자별 자격증명 주입은 contextvar로 스레드별 격리:
  api_key -> `ANTHROPIC_API_KEY`+`--bare`, oauth_token -> `CLAUDE_CODE_OAUTH_TOKEN`, 미설정 -> 전역 폴백.
- 관련 파일: `src/web/auth.py`(`_cred_cipher`/`_enc_secret`/`_dec_secret`/`migrate_encrypt_credentials`)

### 3.4 TLS 종단 (`ab5d654`, `1acc539`, `98c6fcc`)

평문 HTTP에서 로그인 비번/JWT/Anthropic 키가 LAN 스니핑으로 탈취되는 위험을 차단.

- Caddy 리버스 프록시가 HTTPS 종단(자체 CA). meetscript는 compose 내부 `expose`만,
  **평문 8088 포트를 호스트/LAN에 직접 노출하지 않음**. 호스트 `HOST_PORT`(기본 49152) -> `caddy:443`.
- 부수효과: 브라우저 보안 컨텍스트 충족 -> 마이크 녹음(getUserMedia)이 LAN IP 접속에서도 동작.
- **raw IP 접속 대응**: `https://<IP>:포트`는 SNI가 없어 `tls internal`이 인증서를 못 골라
  핸드셰이크가 실패(호스트 121.125.78.171에서 재현). -> `deploy/gen-cert.sh`가 `SITE_HOST`를
  SAN에 박은 self-signed 인증서를 생성, Caddy에 명시적으로 물림(`auto_https off`).
- 업로드 본문 상한 `max_size 200MB`로 과대 페이로드 차단.
- 관련 파일: `deploy/Caddyfile`, `deploy/gen-cert.sh`, `deploy/docker-compose.yml`

### 3.5 배포 하드닝

- **이미지 버전 핀**: `caddy:2.8-alpine` 고정(부동 태그 금지) — 무인 갱신으로 기동 깨짐 방지.
- **모델 read-only 마운트**: `MODELS_DIR:/app/models:ro`, `prompts:/app/prompts:ro`.
- **DB host-dir bind**: `${DATA_DIR}:/app/output` — DB가 호스트 평범한 파일로 떨어져 백업/점검 용이.
- **자원 고갈 완화**: STT 동시성 세마포어 + 큐(초과 요청 대기, `88722da`).
- **원샷 부트스트랩**: `deploy/bootstrap.sh`로 모델/DB 복사 + 빌드/기동 + 헬스체크.

## 4. 보안 점검(2026-06-16)에서 식별된 위험과 조치 현황

위협 모델 = 신뢰된 사내 LAN. 점검에서 이미 견고하다고 평가된 부분과, 식별된 위험의 조치 상태는 다음과 같다.

**이미 견고(유지):** pbkdf2 해시 / `JWT_SECRET` 없으면 부팅 실패 / 전 `/api` 보호 /
회의 owner 토큰강제 + 남의 것 404 은닉 / subprocess argv 리스트(셸 미경유, 인젝션 없음) /
ContextVar 자격증명 격리 / 시크릿 API/로그 미노출 / STT 세마포어 백프레셔.

| 심각도 | 위험 | 조치 | 상태 |
|--------|------|------|------|
| HIGH | HTTP 평문 전송(비번/JWT/키 스니핑) | Caddy TLS 종단(자체 CA) | 완료 |
| HIGH | 공용 약비번 전원 방치 | 초기 비번 강제변경 게이트 | 완료 |
| HIGH | Claude 자격증명 평문 저장 | Fernet at-rest 암호화 | 완료 |
| MED | 로그인 무차별대입 방어 없음 | 실패 카운터/잠금 또는 nginx limit_req | 미적용(후속) |
| MED | 컨테이너 root 실행 | non-root USER | 미적용(후속) |
| MED | 업로드 크기 제한 | 프록시 max_size 200MB + len 체크 | 부분(프록시단 적용) |
| LOW | JWT 폐기 불가 + TTL, 비번변경해도 토큰 유효 | TTL 단축 + token_version | 미적용(후속) |
| LOW | CORS allow_origins=["*"], /api/health 무인증 | CORS 좁히기 | 미적용(후속) |

## 5. 남은 위험 / 후속 과제

1. **무차별대입 방어**: 로그인 실패 카운터+잠금 또는 프록시 rate limit.
2. **컨테이너 non-root 실행**: Dockerfile에 USER 추가.
3. **JWT 폐기 메커니즘**: token_version으로 비번 변경 시 기존 토큰 무효화 + TTL 단축.
4. **CORS 범위 축소**: allow_origins를 실제 출처로 제한.
5. **자체서명 인증서 신뢰**: 클라이언트에 루트 CA 신뢰 등록 안내(첫 접속 경고 제거).
6. **`--bare` 호환성 실검증**: 실 API키로 `--bare` + 시스템프롬프트/슬래시 비활성 호환 확인.

## 6. 배포 운영 체크리스트 (`.env.deploy`)

배포 전 다음 시크릿/설정을 반드시 교체한다(예시값 사용 금지).

- `WEB_AUTH_USERS=admin:<강한비번>,dev:<강한비번>` — 초기 시드(교체 필수)
- `WEB_AUTH_ADMINS=admin` / `WEB_AUTH_PRUNE=0`(일괄등록 계정 보존)
- `JWT_SECRET=<랜덤 hex>` — 미설정 시 부팅 실패
- `CRED_ENC_KEY=<Fernet 키>` — 분실 시 자격증명 복구 불가, JWT_SECRET과 함께 안전 보관
- `SITE_HOST=<접속 IP/호스트명>` — 변경 시 `gen-cert.sh` 재실행 후 caddy 재기동
- `HOST_PORT=49152`(HTTPS), `MODELS_DIR`, `DATA_DIR`
- 절차: `gen-cert.sh`(인증서 생성) -> `bootstrap.sh`(빌드/기동/헬스체크)

> 현재 운영 DB(개발 컨테이너)에는 admin/dev 시드 비번이 약한 기본값으로 남아 있고,
> `must_change_password` 마이그레이션은 security-hardening 계열을 배포해야 적용된다.
