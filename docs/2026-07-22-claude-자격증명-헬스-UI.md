# claude 자격증명 헬스(만료/폐기 사전 감지) — 백엔드 구현 + 프론트 스펙

작성일: 2026-07-22

## 배경 (왜 필요한가)

라이브(`https://lbnote.litbig.com:49152`)에서 "회의 분석은 됐는데 요약이 안 됨" 사고가 반복된다.
근본 원인은 **요약/추출 백엔드가 `agent_cli`(claude 서브프로세스)** 라, 사용자별 claude 자격증명이
없거나(미설정) 만료·폐기되면 전사(Cohere)는 되지만 요약/액션만 조용히 실패하는 구조다.

기존 UI의 한계:

- 설정의 "연결됨" 표시는 **토큰이 저장돼 있다는 뜻일 뿐 유효성은 모른다.** oauth_token(구독 토큰)은
  비밀번호 변경·폐기로 조용히 죽는다.
- 유효성 검증은 **저장하는 순간(PUT) 딱 1회** 뿐. 이후 만료는 회의를 돌려 잡이 실패해야만 드러난다.

목표: **만료/폐기 상태를 UI에서 사전에 보여주고 재인증을 유도**한다. 감지 방식은 **주기적 백그라운드
체크**(서버가 주기적으로 실제 ping) + 설정 화면의 온디맨드 재검증 + 잡 실패 시 사후 배너를 함께 쓴다.

## 상태 모델 (핵심)

| reason | valid | 의미 | 사용자 조치 |
|---|---|---|---|
| `not_configured` | `null` | 자격증명 미설정(행 없음) | 최초 등록 |
| `api_key` | `true` | API 키 — 만료 개념 없음(호출 없이 통과) | 없음 |
| `ok` | `true` | oauth_token 실제 ping 성공 | 없음 |
| `verify_failed` | `false` | oauth_token 만료·폐기(ping 실패) | **재인증(재발급)** |
| `decrypt_failed` | `false` | 행은 있으나 복호 실패(`CRED_ENC_KEY` 불일치/손상) | **재등록**(전체면 서버 키 점검) |
| `unchecked` | `null` | 아직 스윕 전(부팅 직후) | 대기 or 온디맨드 검증 |

> `decrypt_failed` 가 **여러 사용자에 걸쳐 나오면** 개별 만료가 아니라 재배포 때 `CRED_ENC_KEY`가
> 바뀐/빠진 것이다(전체 자격증명 복호 불가). 이건 운영자 서버 조치 대상이다.

## 백엔드 (이 저장소, 구현 완료)

파일: `src/web/app.py`, `src/web/auth.py`. 테스트: `tests/test_credential_health.py`(8케이스).

### 1) `GET /api/settings/claude-credential` (기존 + 확장)

현재 사용자 상태. **캐시된** health 를 함께 반환(실시간 호출 없음 — 가벼움).

```json
{
  "configured": true,
  "type": "oauth_token",
  "updated_at": "2026-07-01T09:12:00",
  "health": { "valid": false, "reason": "verify_failed", "detail": "인증 실패: ...", "checked_at": "2026-07-22T08:00:12" }
}
```
- `health` 는 마지막 스윕/verify 캐시. **아직 검증 전이면 `null`.**

### 2) `GET /api/settings/claude-credential/verify` (신규)

현재 사용자 자격증명을 **실제 claude 호출로 즉시 재검증** + 캐시 갱신. 응답은 (1)과 동일 형태이며
`health` 가 항상 채워진다. oauth_token 이면 호출 1회 발생(설정 화면의 "지금 확인" 버튼용).

### 3) `GET /api/admin/claude-credential-health` (신규, admin 전용)

운영자용 집계 — "1명 vs 전체" 영향 범위 판단. 비관리자 403 / 미인증 401. secret 없음.

```json
{
  "usesAgentCli": true,
  "checkEnabled": true,
  "intervalSec": 21600,
  "lastSweepAt": "2026-07-22T08:00:30",
  "counts": { "configured": 56, "valid": 40, "invalid": 3, "decrypt_failed": 0, "unchecked": 13, "api_key": 5, "oauth_token": 51 },
  "users": [ { "username": "kim", "type": "oauth_token", "updated_at": "...", "valid": false, "reason": "verify_failed", "checked_at": "..." } ]
}
```
- `users` 는 문제(valid=false/null) 우선 정렬 → 운영자가 상단에서 바로 파악.

### 4) 주기 백그라운드 스윕

- 켜짐: 요약/추출이 `agent_cli` 이고 `WEB_CRED_HEALTH_CHECK_ENABLED != 0` 일 때만(lifespan 등록).
- 부팅 후 `WEB_CRED_HEALTH_INITIAL_DELAY_SEC`(기본 60s) → 이후 `WEB_CRED_HEALTH_INTERVAL_SEC`(기본
  21600s=6h) 주기. api_key 는 호출 없이 통과, oauth_token 만 실제 ping. 사용자 간 소량 텀(기본 0.5s).
- 결과는 인메모리 캐시(재기동 시 비고 → 다음 스윕이 복원). audit 이벤트 `cred_health.sweep`.

### 5) 잡 실패 계약(기존, 그대로 활용)

요약 잡이 인증 문제로 죽으면 잡 결과에 `error_code: "claude_auth_expired"` 가 실려 온다
(`GET /api/ai/jobs/{jobId}`). 프론트는 이걸로 **사후 배너**를 띄운다(스윕 주기 사이의 만료도 즉시 포착).

## 프론트 스펙 (LB_Note-web, 별도 저장소에서 구현)

### A. 설정 화면 — claude 자격증명 카드

1. 진입 시 `GET /api/settings/claude-credential` 로 `configured`+`health` 표시.
2. 배지(위 상태 모델 매핑):
   - 미설정: 회색 "미설정" + "등록" 버튼.
   - `api_key` / `ok`: 초록 "정상"(+ `checked_at` "N분 전 확인").
   - `verify_failed`: 빨강 "만료됨 — 재인증 필요" + **[재인증]** CTA(재발급 토큰 입력 폼 오픈).
   - `decrypt_failed`: 빨강 "자격증명 손상 — 재등록 필요"(문구: 서버 키 변경 가능성, 관리자 문의 안내).
   - `null(unchecked)`: "확인 중" + **[지금 확인]** 버튼(→ `.../verify` 호출).
3. **[지금 확인]** = `GET .../claude-credential/verify` 호출(스피너), 응답 health 로 배지 갱신.
4. 저장(PUT) 성공 후에도 응답 `verification.ok` 로 즉시 배지 반영(기존 계약 유지).

### B. 전역 배너 (앱 셸)

- 트리거: (1) 로그인 후 `GET /api/settings/claude-credential` 의 `health.valid === false`,
  또는 (2) 회의 분석/재요약 잡 폴링 결과 `error_code === "claude_auth_expired"`.
- 표시: 상단 고정 경고 배너 "claude 인증이 만료되어 요약/액션이 생성되지 않습니다."
  + **[재인증]** 버튼(설정의 자격증명 카드로 딥링크). 닫기 가능하되 미해결이면 다음 진입 시 재노출.
- 해소: verify 가 `valid===true` 로 돌아오면 자동 숨김.

### C. 관리자 콘솔 — 자격증명 헬스 패널(선택, 권장)

- `GET /api/admin/claude-credential-health` 로 counts 요약 카드(정상/만료/손상/미확인) + 사용자 표.
- 사고 시 "1명인가 전체인가"를 여기서 즉시 판별(특히 `decrypt_failed` 다수 = 서버 키 이슈).

### 카피 (한글, ASCII-safe 유지)

- 정상: `claude 연결 정상`
- 만료: `claude 인증 만료 - 재인증이 필요합니다`
- 손상: `claude 자격증명을 복호화하지 못했습니다 - 재등록이 필요합니다(관리자 문의)`
- 배너: `claude 인증이 만료되어 요약/액션이 생성되지 않습니다. 재인증 후 재요약하세요.`

## 운영 노트

- 이 배포는 전역 claude 키를 넣지 않는다(사용자별). 따라서 `/api/health` 의 전역 `claude_auth`는
  설계상 `no_credentials` 일 수 있고, 그것만으로 장애가 아니다 — 사용자별 health 가 진짜 신호다.
- 관련: `docs/2026-07-08-session-handoff-regenerate-auth-ux.md`(재인증 UX 최초 핸드오프),
  메모리 `web-server-claude-path-gotcha`(sudo 기동 PATH 함정), `regenerate-auth-ux-handoff`.
- env 요약: `WEB_CRED_HEALTH_CHECK_ENABLED`(기본 1), `WEB_CRED_HEALTH_INTERVAL_SEC`(21600),
  `WEB_CRED_HEALTH_INITIAL_DELAY_SEC`(60), `WEB_CRED_HEALTH_PER_USER_DELAY_SEC`(0.5).
