## 요구사항 착수 판단 — 발견 이슈가 걸림돌인가 & 우선순위

### 결론: 하드 블로커 없음
발견된 문제 대부분(반응형·카피·컬러·컨트롤·접근성)은 요구사항 작업과 **영역이 달라 서로를 막지 않는다.** 구조도 요구사항 B(다중 파일)를 무리 없이 받는다.
- 처리 흐름: 프론트 `processMeetingAudio(audioBase64)` → `POST /api/ai/process`(단일) → 백그라운드 잡 → STT. **staging 업로드는 이미 파일당 토큰**(`/api/meetings/audio/staging`)이라 N개 확장이 자연스럽다.
- 병합 난이도 낮음: transcript timestamp가 `MM:SS`이고 초 변환·구간 헬퍼(`app.py` `_ts_to_seconds`, `_segments_from_transcript`)가 **이미 존재** → 파일별 STT 후 누적 길이만큼 offset 더해 잇는 C안이 그대로 붙는다. 신규 기능이지만 **범위가 갇혀 있고 위험 낮음**. (진짜 무게는 UI 이슈가 아니라 이 백엔드 병합이다.)

### 같이 처리해야 할 교차점 (선행 아님, 해당 작업 안에서 동시에)
1. **P0-2 중복 key** — 새로 만드는 다중 파일 리스트는 처음부터 안정적 고유 key로(버그 물려받지 말 것).
2. **P0-1 반응형** — 새 업로드 UI는 반응형으로 작성(전역 사이드바 수정과 별개로 부채 누적 방지). `FileUploadView`는 이미 `md:` 그리드가 있어 결 맞추기 쉬움.
3. **#8 저장 버튼 ↔ 요구사항 A 충돌 주의** — "버튼 합치기/제거" 금지. 자동저장 경로 유지 + 배지·라벨만.

### 우선순위
| 순위 | 항목 | 이유 | 공수 |
|---|------|------|------|
| **1** | **#8 자동저장 배지·라벨**(요구사항 A 마무리) | 유실 사고의 심리적 원인 즉시 제거. 기능은 이미 있어 **표시만** 손봄 = 최고 가성비 | 소 |
| **2** | **#10 다중 파일 업로드**(요구사항 B) | 회의가 2~3파일로 쪼개지는 **실사용 차단**. 신규지만 범위 갇힘. 2단계(프론트 UI → 백엔드 병합) 릴리스 | 프론트 중 / 백 중~대 |
| **3** | **#1 반응형** + **#2 중복 key(전역)** | 사용성/버그 직결. 전역 레이아웃이라 **별도 담당이 병렬** 진행 가능 | 중 |
| **4** | **#3·4·5 카피/컬러/컨트롤** | 공수 대비 완성도 체감 큼, 여러 화면 동시 개선 | 소~중 |
| **5** | **#6·7 통계라벨·멤버데이터 → #9 접근성** | 정합성·품질 마무리 | 중 |

> 핵심: #8은 퀵윈이라 먼저 털고, #10을 이어서. P0(반응형/키)은 요구사항과 독립이라 **병렬 트랙**으로 돌리면 서로 안 막힌다. **선행으로 반드시 고쳐야 요구사항이 통과되는 문제는 없다.**

## 🔴 P0 — 반드시

### 1. 모바일/반응형 완전 미대응
**문제.** 사이드바가 고정 폭이라 모바일에서 본문이 화면 밖으로 밀리고 제목이 겹쳐 잘린다. 햄버거/드로어가 없어 390px에서 사실상 사용 불가.

**근거.** `App.tsx:500`
```tsx
<aside className="w-80 relative border-r ... flex flex-col ...">
```
`w-80`(320px) 고정 + 브레이크포인트 클래스(`lg:`, `md:hidden`) 전무.

**개선 방법.**
1. 데스크톱은 고정, 모바일은 오버레이 드로어로 분리.
   ```tsx
   // 상태 추가
   const [navOpen, setNavOpen] = useState(false);

   // 사이드바: lg 이상 고정, 그 이하는 슬라이드 드로어
   <aside className={`
     fixed inset-y-0 left-0 z-40 w-80 bg-bg-primary border-r border-border-primary
     flex flex-col transition-transform duration-300
     lg:static lg:translate-x-0
     ${navOpen ? 'translate-x-0' : '-translate-x-full'}
   `}>
   ```
2. 모바일 전용 상단 앱바 + 햄버거 추가(로고 + 메뉴 토글), `lg:hidden`.
3. 드로어 열렸을 때 배경 딤(`fixed inset-0 bg-black/40 lg:hidden`), 클릭 시 닫기.
4. `main`은 `flex-1 min-w-0`로 두어 좁은 폭에서 내부가 잘리지 않게(현재 `overflow-hidden`이 콘텐츠를 감춤).

**검증.** Playwright 390 / 768 / 1440px 3폭 스크린샷으로 잘림·겹침 없음 확인.

### 2. React "중복 key" 경고 다수
**문제.** 콘솔에 `Encountered two children with the same key` 가 여러 리스트에서 반복. key 중복은 항목 누락/중복 렌더·상태 꼬임으로 이어지는 실제 버그.

**개선 방법.**
- 회의·액션·참석자·전사 세그먼트 등 `.map()` 렌더의 `key`를 **안정적 고유 id**로 교체. 배열 인덱스나 중복 가능한 필드(예 같은 제목/빈 id) 사용 금지.
- 신규 항목은 생성 시점에 고유 id 부여(`crypto.randomUUID()` 등). 서버 id가 없으면 임시 uid를 만들어 저장 후 교체.
- 빠른 진단: dev 콘솔의 경고에 찍히는 컴포넌트 스택으로 대상 리스트 특정 → 해당 `key=` 수정.

**근거(예).**
- `components/ActionExtractionView.tsx:172` — `CHARS`(글자수), `CLEAR`, `UPLOAD FILE`
- `components/Dashboard.tsx:333` — `ACTION PLANNING`

- 브랜드성으로 영문을 남길 섹션 타이틀이 있으면, **한 규칙**으로 통일(예: 섹션 kicker만 영문 소문자 tracking, 본문/버튼은 전부 한글).

### 4. 브랜드 컬러 이탈 — 액션추출 CTA만 초록
**근거.** `components/ActionExtractionView.tsx:179` — 주 버튼 `bg-green-600`. 다른 화면 primary는 오렌지(`bg-orange-600`).

### 5. 네이티브 select/date 컨트롤 불일치 · 날짜 포맷 혼재
**문제.** 상태 드롭다운·날짜 입력이 브라우저 기본 룩이라 커스텀 라운드 디자인과 이질적. 날짜 포맷도 화면마다 다름(파일업로드/리뷰의 네이티브 date는 미국식 `MM/DD/YYYY`, 대시보드는 `7/7/2026`, 리뷰 헤더는 `12:00:00 AM`).

**근거.** 네이티브 `<select>`: `ActionItemsView / CalendarMeetingsView / IntegrationSettingsView / ParticipantsView / AdminUserManagement / RecorderView / ReviewView`. 네이티브 `type="date"`: `CalendarMeetingsView.tsx:714,728,744,771`, `FileUploadView.tsx:312`, `RecorderView.tsx:351`.

**개선 방법.**
- 공용 `Select`/`DatePicker` 컴포넌트로 교체(라운드·보더·포커스링을 디자인 토큰에 맞춤). 최소한 `<select>`에 커스텀 chevron + `appearance-none` + 공통 클래스라도 적용.
- 날짜/시간 표기를 한국 로케일 유틸로 통일: 표시 `2026.07.09`, 24시간 `14:30`. `new Intl.DateTimeFormat('ko-KR', ...)` 기반 `formatDate()/formatTime()` 헬퍼를 만들어 전역 사용.
- 입력용 네이티브 date가 필요하면 최소한 표시 텍스트는 한국식으로 재가공.

### 7. 멤버 인사이트 "0명" — 데이터 소스 불일치
**문제.** 실제 사용자 56명인데 멤버뷰는 "0명 등록됨". 인증 디렉터리가 아니라 브라우저 `localStorage` stub를 소스로 쓰기 때문.
**근거.** `services/memberService.ts` — `const KEY='lbnote.members'`, `localStorage` 기반(주석에 "v2에서 백엔드 `/api/members`로 승격 가능"). 반면 실제 명부는 `/api/directory`(56명).
**개선.**
- `memberService.getMembers()`를 `/api/directory`(또는 신규 `/api/members`)에 연결해 실제 사용자와 통계를 채운다.
- 발화 점유율/할당 과제 같은 인사이트는 회의·액션 데이터에서 집계(백엔드 집계 엔드포인트 or 프론트 조인).
- 과도기라면 최소한 "로컬 저장이라 비어 보임"을 빈 상태 문구로 정직하게 안내.

### 8. 리뷰 상단 저장 버튼 2개 모호 — 자동저장과의 관계 불명확
> ⚠️ **요구사항 A와 직결.** 자동 저장(요약 시)은 이미 동작하며, 두 버튼도 그 후속(변경 업데이트 / 확정)이라 **의도적**이다. 따라서 "버튼을 합치는" 방향이 아니라, **자동저장됨을 알리고 각 버튼의 역할을 드러내는** 방향으로 개선한다.

**문제.** 요약 직후 이미 자동 저장됐는데도 화면상 그 사실이 약하게만 보이고, 상단의 "업데이트 저장"·"저장"(및 인라인 "요약 저장"/"전사 저장")의 차이가 라벨만으로 불명확 → 사용자가 여전히 "저장해야 하나?"로 혼란(요구사항 A의 유실 사고 심리와 동일 지점).

**근거.** `components/ReviewView.tsx`
- `handleSave`(244) → "변경 사항을 저장했습니다" = **업데이트 저장**(초안 갱신)
- `handleFinalize`(256) → `status:'sent'` + Drive 동기화 = **확정 저장**
- `handleSaveTranscript`(272) 등 부분 인라인 저장 별도.

**개선.**
- **자동저장 상태 배지**: 헤더에 "자동 저장됨 · HH:MM"(마지막 저장 시각)을 상시 노출 → 유실 불안 제거.
- **라벨을 동작 기준으로**: "업데이트 저장" → **"변경사항 저장"**, "저장"(확정) → **"확정 · 발송 준비"**(status='sent' 의미를 드러냄). 인라인은 "요약만 저장/전사만 저장"으로 범위 명시.
- 버튼 위계: primary(오렌지) 1개(확정) + secondary(보더) 1개(변경 저장)로 시각 구분.
- **버튼을 없애거나 합치지 말 것** — 자동저장(요구사항 A) 후 수동 업데이트/확정 경로는 유지해야 한다.

## 🟢 P3 — 접근성(a11y)

### 9. label/aria/landmark 부재
**문제.** 폼 입력이 `placeholder`만 있고 `<label>`/`aria-label` 없음(로그인·검색·참석자 추가 등). 랜드마크가 `nav/main/header` 대신 `div` 위주라 스크린리더·자동완성이 취약. 로그인 disabled 버튼이 연한 오렌지라 활성처럼 보임.
**개선.**
- 모든 입력에 연결된 `<label>` 또는 `aria-label` 부여(시각적으로 숨기려면 `sr-only`).
- 시맨틱 태그로 감싸기: 상단 `header`, 좌측 `nav`(이미 `nav` 사용부는 유지), 본문 `main`(현재 `main` 있음 — 나머지 랜드마크 보강).
- 비활성 버튼은 회색 계열로(`disabled:bg-border-primary disabled:text-text-secondary`) 상태를 명확히. 오렌지 위 흰 텍스트 등 대비 4.5:1 점검.

## 🟠 P1(요구사항 B) — 다중 파일 업로드(한 회의 = 파일 2~3개)

**문제.** 같은 회의가 1·2차로 나뉘거나 녹음이 끊겨 파일이 2~3개 생기는데, 현재 파일 업로드는 **단일 파일 전용**이라 한 회의로 합쳐 분석할 수 없다.

**근거.** `components/FileUploadView.tsx`
- 단일 상태: `const [file, setFile] = useState<File | null>(null)`(34)
- 단일 취득: `handleChange` `e.target.files[0]`(94), `handleDrop` `e.dataTransfer.files[0]`(88)
- input에 `multiple` 없음(250-256), 업로드/처리 흐름(`handleUpload` 120-175)이 파일 1개 전제.

**개선 방법 (프론트).**
1. **다중 선택 UI**: input에 `multiple` 추가, 상태를 `useState<File[]>([])`로. 드롭/선택 시 배열에 누적.
   ```tsx
   <input type="file" multiple accept="audio/*,.wav,.mp3,.m4a,..." onChange={handleChange} />
   // handleChange: [...files, ...Array.from(e.target.files)]
   // handleDrop:   [...files, ...Array.from(e.dataTransfer.files)]
   ```
2. **순서 지정 리스트**: 업로드된 파일을 카드 리스트로 보여주고 **드래그 재정렬 + 개별 삭제 + 순번(1차·2차…)** 표시. 녹음이 이어지는 회의라 **순서가 곧 시간축**이므로 순서 UI가 핵심.
3. **합산 정보**: 총 길이/총 용량 표기, "N개 파일을 하나의 회의로 분석합니다" 안내.

**개선 방법 (처리 파이프라인 — 3안, 권장 C).**
- **A. 프론트 concat**: 브라우저에서 오디오를 이어붙여 단일 blob → 비권장(디코드/리샘플 복잡, 큰 파일 메모리 부담).
- **B. 백엔드 concat 후 단일 STT**: staging 토큰 N개를 순서대로 넘기면 서버가 이어붙여 1회 STT. 타임스탬프 연속성 좋음. 서버 작업 필요(디코드·리샘플·concat).
- **C. 파일별 STT 후 transcript 병합(권장)**: 각 파일을 기존 `vad_chunk` 파이프라인(`/app src/pipeline.py`)으로 독립 처리 → transcript를 **순서대로 이어붙이되 앞 파일 누적 길이만큼 timestamp offset 가산**. 요약·액션은 병합된 전체 transcript로 **1회** 생성.
  - 온프레미스 STT 구조(파일 단위 디코딩)와 가장 잘 맞고, 파일 경계에서 seam dedup만 신경 쓰면 됨.
  - 서버 계약(예): `POST` 처리 요청에 `audioStagingTokens: string[]`(순서 보존) → 각 토큰 STT → offset 병합 → 단일 회의 결과 반환. 원본 N개는 Drive에 `원본_1.ext … 원본_N.ext`로 보관.

**주의/엣지케이스.**
- 파일 순서 = 시간 순서. 사용자가 정한 순서를 **서버까지 그대로 전달**(정렬 재계산 금지).
- 서로 다른 샘플레이트/코덱 혼합 가능 → 서버에서 표준 포맷으로 정규화 후 병합.
- 부분 실패(3개 중 1개 STT 실패) 시: 성공분만 병합 + 어떤 파일이 빠졌는지 명시(요구사항 A의 "조용히 유실 금지" 원칙과 동일).
- 실시간 녹음 쪽(`RecorderView`)에도 동일 니즈("녹음 끊김 후 이어녹음")가 잠재 → 같은 병합 계약을 재사용하도록 설계.

**예상 공수.** 프론트(다중 선택·순서 UI) 중 / 백엔드(멀티 토큰 STT·offset 병합) 중~대. 프론트 UI를 먼저 내고, 처리로 C안을 붙이는 2단계 릴리스 권장.
