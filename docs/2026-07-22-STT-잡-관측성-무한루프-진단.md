# STT/요약 잡 관측성 — "왜 무한루프인지" 진단 정보 제공

작성일: 2026-07-22

## 문제

사용자가 STT를 돌릴 때 스피너가 안 끝나는("무한루프") 경우, **원인이 셋인데 구분할 신호가 없었다.**

| 원인 | 실제 상태 | 근거 |
|---|---|---|
| 다른 사람이 사용 중(경합) | 잡 `queued` 지속 | STT 동시성 기본 1(직렬). 슬롯 1개라 앞 잡이 안 끝나면 뒤는 대기 |
| 백엔드 엔진 이상(진짜 무한) | 잡 `processing` 무한 | `transcribe_to_segments`(STT)에 타임아웃·취소지점 없음 → 멈추면 슬롯 영구 점유 → 전원 무한대기 |
| 자격증명 문제 | STT 무관, 요약만 실패 | STT는 자격증명 미사용. 인증 문제는 요약 단계 `error_code=claude_auth_expired` |

기존 폴링(`GET /api/ai/jobs/{id}`)은 `queued/processing/done/error` 4개만 줘서 원인 표시가 불가능했다.

## 백엔드 (이 저장소, 구현 완료)

파일 `src/web/app.py`. 테스트 `tests/test_job_observability.py`(7케이스). 전체 298 통과·ruff 클린.

### 1) 잡 진행 phase + 타이밍

STT 잡은 `waiting_stt → transcribing → waiting_llm → summarizing` 단계를 기록한다(재요약 잡은
`waiting_llm → summarizing`). 각 잡에 `created_at/started_at/phase_at`(monotonic)을 남겨 경과를 계산.

### 2) `GET /api/ai/jobs/{id}` 응답 확장 (기존 필드 + 아래)

```json
{
  "jobId": "…", "status": "processing",
  "phase": "transcribing",
  "elapsedSec": 12.4, "phaseElapsedSec": 12.4,
  "warning": null,
  "queue": { "sttSlots": 1, "sttActive": 1, "sttQueued": 2, "llmSlots": 2, "llmActive": 0, "llmQueued": 0 },
  "ahead": 2,
  "reasonHint": "전사(STT) 진행 중입니다."
}
```
- `phase`: 무엇을 기다리는지/처리 중인지.
- `queue`: 슬롯 점유·대기 집계 → **"누가 쓰고 있어서"인지**(sttActive>0, sttQueued>0) 판별.
- `ahead`: STT 대기 잡의 내 앞 처리 건수(경합 정도).
- `warning`: `stt_stalled`(스톨 탐지 시).
- `reasonHint`: 프론트가 그대로 노출 가능한 한 줄 원인(경합/전사중/요약중/스톨/엔진오류/인증만료).

### 3) error_code 분류 (원인별)

- `claude_auth_expired` — 자격증명(요약/추출 단계). → 재인증.
- `stt_engine_error` — STT 단계 예외(전사 실패). → 백엔드 엔진 점검.
- `stt_stalled` — STT 스톨 워치독 마감(옵션). → 엔진 점검/재기동.

### 4) STT 스톨 소프트 워치독

- `WEB_STT_STALL_SCAN_SEC`(기본 30s) 주기로 스캔. `phase=transcribing` 이 `WEB_STT_STALL_SEC`
  (기본 900s=15분)를 넘으면 스톨로 판정 → `warning=stt_stalled` + audit `stt.stall` + stdout 로그.
- **소프트**: 워커/토치 추론을 강제 종료하지 않는다(안전). 슬롯 회수는 프로세스 재기동으로만.
  로그가 운영자에게 "이 슬롯이 물렸다"를 알린다.
- `WEB_STT_STALL_MARK_ERROR=1` 이면 대기 사용자에게 `error(stt_stalled)`로 마감까지(무한 스피너 종료).
  기본 OFF — 정품 장시간 STT 오탐 위험. 임계값은 RTFx≈232라 정상은 분 단위 이하여서 넉넉하다.

### 5) `GET /api/admin/ai-jobs` (admin 전용)

진행 중 잡/슬롯 스냅샷 — 경과시간 내림차순(오래 걸린 잡·스톨 상단). 운영자가 "경합 vs 엔진 스톨"을
즉시 판별.

```json
{ "queue": {…}, "sttStallSec": 900, "markError": false,
  "activeCount": 3, "stalledCount": 1,
  "active": [ { "jobId": "…", "owner": "kim", "kind": "stt", "status": "processing",
               "phase": "transcribing", "warning": "stt_stalled", "elapsedSec": 930.2, "phaseElapsedSec": 930.2 } ] }
```

## 프론트 스펙 (LB_Note-web, 별도 저장소)

### A. 분석 진행 화면(폴링 UI)

폴링마다 `reasonHint`/`phase`/`queue`/`elapsedSec` 로 상태 문구를 바꾼다:

- `queued` + `ahead>0`: **"앞선 처리 N건 대기 중"**(경합) — 스피너 유지, 예상 대기 안내.
- `processing` + `transcribing`: "전사 중 (경과 Ns)".
- `processing` + `summarizing`: "요약/추출 중".
- `warning=stt_stalled`: **"전사 엔진이 응답하지 않습니다(경과 Nm). 관리자 확인이 필요합니다."**
  — 스피너를 경고 상태로 바꾸고 [취소] 제공. **자동 재시도 금지**(막힌 큐에 잡만 쌓임).
- `error` + `error_code`별:
  - `claude_auth_expired` → 재인증 배너/CTA(자격증명 헬스 UI와 연동, docs/2026-07-22-claude-자격증명-헬스-UI.md).
  - `stt_engine_error`/`stt_stalled` → "백엔드 엔진 오류 — 잠시 후 재시도/관리자 문의".

### B. 관리자 콘솔 — 잡 모니터(선택, 권장)

`GET /api/admin/ai-jobs` 로 슬롯 점유(sttActive/sttQueued)·활성 잡 표·스톨 배지. 사고 시
"1명 경합인지 엔진 스톨인지"를 여기서 판별.

## 운영/후속

- 이건 **관측·조기경보**다. STT 슬롯을 실제로 회수하려면 재기동이 필요하다(소프트 워치독의 한계).
  근본 차단(서브프로세스 격리+하드 타임아웃으로 멈춘 추론 kill)은 별도 과제로 남긴다 — 트레이드오프는
  구현 규모와 정품 장시간 STT 오탐이다.
- STT 병목 완화는 `WEB_STT_CONCURRENCY` 상향(VRAM 여유 시)도 고려.
- env: `WEB_STT_STALL_SEC`(900), `WEB_STT_STALL_SCAN_SEC`(30), `WEB_STT_STALL_MARK_ERROR`(0).
