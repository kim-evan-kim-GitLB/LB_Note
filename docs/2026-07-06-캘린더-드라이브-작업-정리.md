## 3. 다음 작업 — Google Drive 저장 구조 (설계 확정, 미착수)

### 배경: 두 파이프라인은 사실상 동일
사전 녹음 업로드와 실시간 녹음 두 경로 모두 `processMeetingAudio` → Meeting → drive-sync 로 수렴한다. 분기점은 둘뿐:
1. 참석자(실시간=화자, 업로드=[]).
2. **원본 미디어 영속화**: 실시간은 audio 를 staging → `audioRef` 바인딩 → Drive 에 오디오 저장됨. 업로드는 staging 안 해서 Drive 에 Doc 만 올라가고 원본이 안 감.
→ 결론: Drive 저장을 Meeting 계층에서 통일하고, 업로드 경로의 미디어 영속화 갭을 닫는다.

### 확정된 요구사항
- 저장 위치: 회사 정책상 공유 드라이브 미정의 → **사용자 개인 "내 드라이브"**.
- 폴더 구조: `LB_NOTE/{회의명_날짜_시간}/{원본.{ext}, 회의록}`.
- 원본은 **무변환**(실제 확장자 그대로 저장, transcode 없음).
- 원본·회의록 **둘 다 저장**(업로드 경로도 원본 미디어 영속화하도록 대칭화).

### 구현 계획
- 백엔드 `src/web/google_drive.py`:
  - `ROOT_FOLDER_NAME` `"회의록 (meetscript)"` → `"LB_NOTE"`.
  - `ensure_subfolder(access, parent_id, name)` 추가(find-or-create).
  - 회의별 하위 폴더 `{sanitized_회의명}_{YYYY-MM-DD}_{HHMM}`(KST). Doc 이름 `회의록`, 오디오 이름 `원본.{ext}`(실제 확장자) 을 하위 폴더 안에 저장.
  - 하위 폴더 id 를 `gdriveRef.folderId` 에 저장 → 재싱크 시 재사용. 기존 flat 파일은 그대로 둠.
  - `upsert_audio` 는 audio_id 존재 시 skip(불변, resumable ≤500MB).
- 프론트 업로드 대칭화 `FileUploadView.tsx`: 원본 파일 staging → `audioStagingToken` 부착 → `audioRef` → Drive 에 `원본.{ext}` 저장. (`RecorderView.tsx` 의 `uploadAudioStaging` 패턴 참고.)
- 검증: pytest + ruff + tsc.

## 3-bis. 오디오 원본 Drive 오프로드 (구현 완료, 미커밋)

배경: 음성 원본은 무거워(1h ≈ 30~60MB) 서버 디스크를 압박하고, 과거 무백업 데이터 유실 사고도 있었다.
결정(사용자): **원본을 Drive 에 저장하는 걸 원칙**으로 하되, "내보내기 완료까지만 로컬 보관 → 업로드
성공 시 삭제". 업로드는 **저장 시 자동(백그라운드)**. 삭제 후 재생은 **Drive 프록시로 앱 내 유지**.

동작:
1. **저장 시 자동 동기화** — 프론트 `App.tsx trySyncMeetingToDrive` 가 finalize(생성/수정) 직후
   fire-and-forget 으로 drive-sync 잡 시작(캘린더 자동연동과 동일 패턴). 미연동이면 조용히 스킵.
2. **업로드 성공 → 로컬 삭제** — `_run_drive_sync_job` 이 `audioId` 확정 시 `audio_store.
   delete_meeting_audio` 로 로컬 원본 정리. audioRef 메타(format/size)는 DB 에 남는다.
   실패·미연동·인증만료면 삭제 안 함(원본 안 잃음 · Google 강제 아님).
3. **재생 프록시** — `get_meeting_audio` 가 로컬 부재 시 `_stream_audio_from_drive` 로 폴백:
   `gref.audioId` + 사용자 cred → `google_drive.stream_media`(urllib 로 Drive `alt=media` Range GET)
   → status/헤더(206/200/416, Content-Range/Length) 릴레이. 프론트 재생 URL 은 그대로(변경 없음).
4. **토큰 캐시** — 재생은 Range 요청이 재생 1건당 수십 회라, `google_oauth.refresh_access_token_cached`
   (만료 skew 기반 인메모리 캐시)로 토큰 엔드포인트 과호출 방지.

재sync: 로컬이 이미 지워졌으면 upsert_audio 미호출(재업로드 없음), `audioId` 는 gref 에서 보존.

코드리뷰(Critical/High 0) 반영 완료:
- 동시 동기화 디둡: `drive-sync` 엔드포인트에 `_drive_sync_inflight`(meeting_id→job_id) 가드 —
  자동+수동/연속 저장이 같은 회의에 겹치면 기존 잡 재사용(중복 폴더/Doc 레이스 방지).
- 삭제 시 빈 하위폴더 정리: `_delete_google_assets_async` 가 `folderId`(≠루트)면 폴더째 삭제
  (내용 동반). 루트와 같으면(구 flat 회의) 개별 파일만 삭제해 루트 오삭제 방지.
- 프록시 견고화: `stream_media` 가 Message(케이스 무시 헤더) 반환 + 416 시 fd 즉시 close.

후속(비차단): 저장마다 자동 sync 는 편집 Doc 반영 목적상 의도된 동작(디바운스 불요). 대용량
`alt=media` 302 리다이렉트 시 Range/Authorization 보존은 라이브 검증 권장.

검증: 백엔드 240 통과(프록시 릴레이·로컬삭제·재sync 보존·토큰캐시·동시 디둡·하위폴더 삭제 신규
테스트) · ruff clean · tsc OK.
